# Shape, Elevation, Motion, Z-Index

## Radius — three shapes only

export const RadiusDemo = ({ radius, label }) => (
  <span
    style={{
      display: 'inline-block',
      width: '4rem',
      height: '2rem',
      borderRadius: radius,
      border: '1px solid var(--og-line-strong)',
      background: 'var(--og-surface)',
      verticalAlign: 'middle',
    }}
    aria-label={label}
  />
);

| Preview | Token / utility | Value | Use |
|---------|-----------------|-------|-----|
| <RadiusDemo radius="0.375rem" label="control" /> | `rounded-control` | 0.375rem (md) | inputs, selects, inline form buttons, notice boxes in forms |
| <RadiusDemo radius="0.75rem" label="card" /> | `rounded-card` | 0.75rem (xl) | cards, list panels, notices, empty states |
| <RadiusDemo radius="999px" label="pill" /> | `rounded-full` | pill | pill buttons, nav links, chips/badges, dots, progress bars, toggles |

Never `rounded-lg`, `rounded-sm`, bare `rounded`, or an arbitrary radius.
Rule of thumb: **panels are cards, controls are controls, everything small is
a pill.** Standalone action buttons are pills (`rounded-full`); buttons that sit
inside a vertical form stack are controls (`rounded-control`) so they align
with the inputs.

## Elevation

Flat design — borders separate surfaces, not shadows.

| Utility | Use |
|---------|-----|
| (none) | default: cards use `border border-line`, no shadow |
| `shadow-sm` | sticky/overlay elements that float over content |
| `shadow` | the toggle knob only |

No other shadow values, no arbitrary `shadow-[…]`.

## Z-index

None. The app has no modals, dropdowns, or toasts; stacking is handled by
`sticky` table columns without an explicit z-index. If a layered component is
ever added, mint `--z-*` tokens in `app/tokens.css` first.

## Motion

For everyday UI, keep it boring: state changes get a plain transition, default
timing.

| Utility | Use |
|---------|-----|
| `transition-colors` | hover color changes (buttons, links, rows) |
| `transition-transform` | toggle knob |
| `transition-all` | progress-bar width |

In app UI, prefer default duration/easing — avoid `duration-*` / `ease-*` on
routine hovers.

### Motion tokens

Richer, character-driven motion (login micro-interactions, future celebrations)
can draw on named tokens in `app/tokens.css` so timing stays consistent. These
are not theme-dependent. The mascot itself is a Lottie — its timing lives baked
in `public/mascot/pott.json`, not in these tokens.

| Token | Value | Use |
|-------|-------|-----|
| `--dur-quick` | 140ms | small hovers, quick fades |
| `--dur-base` | 240ms | field underline, label float |
| `--dur-slow` | 440ms | larger reveals |
| `--ease-out-soft` | cubic-bezier(.22,1,.36,1) | settle-in, no overshoot |
| `--ease-spring` | cubic-bezier(.22,1.28,.36,1) | playful overshoot |

Reference them in component CSS or an inline `style` as `var(--ease-spring)`.

## Personality & mascot

The app has a mascot, **Pott** (an enamel Dutch oven with a face). It carries
the playful, gamified side of the product. It is a real reusable component
rendered as a **Lottie** (`public/mascot/pott.json`, generated by
`scripts/buildPottLottie.mjs`) — documented at
[Components/Mascot](/docs/components-mascot-pott--docs) and, for its role on the
login screen, [Patterns/Login-Erlebnis](/docs/patterns-login-erlebnis--docs).

**Rules of engagement:** on `prefers-reduced-motion: reduce` the mascot holds a
static pose instead of animating (handled in `LottieMascot.tsx`); it is
`aria-hidden` unless given a `title`; and it never blocks or delays a real
interaction — it reacts to state, it doesn't gate it.
